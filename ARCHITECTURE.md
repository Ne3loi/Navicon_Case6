# Архитектура сервиса

## Назначение

`Navicon Sanitizer 3.0` — локальный сервис обезличивания документов перед отправкой во внешний контур.

Сервис построен по схеме `human-in-the-loop`:

1. пользователь загружает документ
2. сервис находит чувствительные данные
3. пользователь подтверждает или снимает отдельные совпадения
4. сервис формирует безопасную версию файла и отчет

## Архитектурные компоненты

### Frontend

Файл: [app.py](app.py)

Функции:

- загрузка одного или нескольких файлов
- выбор категорий поиска
- запуск анализа
- просмотр подсветки и таблицы совпадений
- ручное добавление фраз для вычеркивания
- подтверждение результата
- скачивание итогового ZIP-архива

Технология: `Streamlit`

### Backend API

Файл: [backend/app.py](backend/app.py)

Функции:

- прием файлов
- запуск анализа
- хранение временной сессии анализа
- формирование итогового архива после подтверждения

Технология: `FastAPI`

Основные endpoint:

- `GET /health`
- `POST /analyze`
- `POST /redact/{analysis_id}`

### Ядро анализа

Файл: [backend/core.py](backend/core.py)

Функции:

- извлечение текста из `PDF`, `DOCX`, `TXT`, `MD`
- OCR для `PNG`, `JPG`, `JPEG` и сканов внутри PDF
- поиск сущностей через `Natasha`, `regex` и `Qwen`
- построение подсветки и отчета
- вычеркивание в тексте, PDF и изображениях
- сборка итогового ZIP

## Поток обработки

### 1. Анализ

Frontend отправляет файлы в backend через `POST /analyze`.

Backend:

- распаковывает ZIP-архивы
- определяет формат каждого файла
- извлекает текст или запускает OCR
- находит чувствительные фрагменты
- возвращает frontend:
  - summary
  - hits
  - preview
  - verdict

### 2. Подтверждение

Пользователь:

- оставляет или снимает галочки у найденных совпадений
- при необходимости добавляет ручные фразы для вычеркивания

### 3. Вычеркивание и экспорт

Frontend отправляет подтвержденный набор совпадений в `POST /redact/{analysis_id}`.

Backend:

- формирует очищенные версии файлов
- создает `report.json` и `report.md`
- упаковывает результат в ZIP

## Хранение данных

В текущей версии:

- внешняя база данных не используется
- миграции отсутствуют
- состояние анализа хранится во временном in-memory словаре backend-сервиса
- время жизни сессии ограничено `ANALYSIS_TTL_SECONDS`

Следствие:

- после рестарта backend активные сессии анализа сбрасываются
- для production-версии можно заменить in-memory слой на Redis/PostgreSQL, если потребуется постоянное хранение

## Поддерживаемые форматы

- `PDF`
- `DOCX`
- `TXT`
- `MD`
- `PNG`
- `JPG`
- `JPEG`
- `ZIP`

## Поиск сущностей

### Natasha

Используется в первую очередь для русскоязычных текстов.

### Regex

Используется для детерминированных паттернов:

- email
- телефон
- ИНН
- паспорт
- суммы
- счета
- организационные формы

### Qwen

Может использоваться как внешний LLM-движок через OpenAI-совместимый endpoint, если переменные `QWEN_*` заданы в `.env`.

Если Qwen не настроен, сервис работает без него.

## Инфраструктурная схема

### Локальный запуск

`run.bat -> run_local.py -> backend + frontend`

### Стенд / VM

`nginx -> frontend (Streamlit) -> backend (FastAPI)`

## Выходные артефакты

После обработки пользователь получает ZIP-архив, который может содержать:

- очищенный файл в исходном формате
- `Markdown`
- `Word (.docx)`
- `report.json`
- `report.md`

## Ограничения текущей версии

- для OCR-изображений качество зависит от качества исходного скана
- изображения и scan-only PDF обрабатываются менее стабильно, чем документы с текстовым слоем
- состояние сессии хранится в памяти процесса и не переживает рестарт backend
