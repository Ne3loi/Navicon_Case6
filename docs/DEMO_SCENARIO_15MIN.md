# Демо-сценарий (15 минут) для жюри

## 0. Подготовка за 15-30 минут до показа

1. Убедиться, что сервис открыт по HTTPS: `https://<VM-IP>`.
2. Проверить `docker compose ps` (должны быть `backend`, `frontend`, `nginx`).
3. Подготовить 3 файла для показа:
   - `test/Документ Microsoft Word.pdf`
   - `test/Документ Microsoft Word.docx`
   - `test/img-2026-02-24 15.38.33.png`
4. Открыть интерфейс и прогреть страницу (первый проход может быть медленнее из-за инициализации моделей).

   - что OCR отработал на картинке.
6. Снять 1-2 галочки в таблице (пример: оставить безопасную сущность).
7. Нажать `2. Подтвердить и вычеркнуть`.
8. Скачать `sanitized_results.zip`.
9. Показать содержимое архива:
   - очищенный PDF/Word,

### 13:00-15:00 — Вопросы

## 2. Готовые ответы на типовые вопросы

### "Почему не просто regex?"

- "Regex закрывает часть кейсов, но имена/организации/контекст лучше покрываются NER. Мы используем комбинированный подход."

### "Где хранятся данные?"

- "Обработка локальная. В памяти backend с TTL для сессии анализа, без передачи содержимого в публичные облака по умолчанию."

### "Почему 2 шага, а не сразу удаление?"

- "Это снижает риск ложных срабатываний и дает бизнес-пользователю контроль перед отправкой документа."

### "Готово к масштабированию?"

- "Да. Архитектура сервисная, есть reverse proxy, API-слой, контейнеризация и подключаемые движки."

## 3. План B на случай сбоя во время демо

1. Если OCR медленно стартует: показать сначала PDF+DOCX, затем отдельно PNG.
2. Если нет интернета/внешних сервисов: подчеркнуть, что решение локальное и работает без внешних API.
3. Если браузер ругается на сертификат: заранее принять self-signed сертификат на демонстрационном ноутбуке.
